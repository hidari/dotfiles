# nvim カラーパレットを実効背景基準で再設計する

作成日: 2026-07-10

## 背景

nvim の Markdown ハイライトとファイラ (neo-tree) で、一部の色がターミナル上でまったく見えない。
特に以下が読めない。

- Markdown のマークアップ記号 (`#` マーカー、コードフェンス、表の `|`、リンク URL)
- Markdown の打ち消し線
- neo-tree の gitignore 対象ファイル

## 実測で分かったこと

### 実効背景は固定値ではない

nvim は `Normal` の背景を `NONE` にしてターミナル背景を透過させている。
そのターミナルは Ghostty で、テーマ Mariana (背景 `#343d46`)、`background-opacity = 0.8`。

したがって文字の背後にあるのは Ghostty の背景色ではなく、
それを背後のウィンドウと 0.8 で合成した色になる。

- 背後が黒いウィンドウ: 実効背景は `#2a3138`
- 背後が白いウィンドウ: 実効背景は `#5d646b`

背後が白いときの合成値 `#5d646b` は、実際のスクリーンショットの最頻色 (画面の 35.7%) と完全に一致した。
仮説ではなく実測で裏が取れている。

### 前景を暗くして逃げる余地はない

実効背景は明るい方向にしか動かない。暗い方向には `#2a3138` で止まる。
このため「マークアップ記号を本文より暗くして視覚的に後退させる」という従来の設計は、
透過環境では原理的に成立しない。

決定的なのは、問題の色 `#5c6370` が不透明な Mariana 背景の上でも 1.83:1 しかないこと。
透過をやめても、`background-opacity` を 0.95 に上げても、この色は救えない。
透過は問題を悪化させただけで、根本原因は色そのものが暗すぎることにある。

### neo-tree には gitignored 以外にも穴があった

neo-tree はファイル名の色を決めるとき、filtered (dotfile / gitignored など) の色を先に決め、
そのあと git status の色で上書きする。このため未追跡ファイルは名前そのものが
`NeoTreeGitUntracked` の色で描かれ、これが 2.49:1 だった。

またツリーの薄灰 (インデント線、展開マーカー、`(empty folder)`) は `Comment` ではなく
`NeoTreeDimText` のハードコード値に由来する。`Comment` を直しても治らない。

## 設計原則

### 基準背景は最悪ケースを取る

パレットの設計基準を、背後が白いウィンドウのときの実効背景とする。
ここでコントラストを確保すれば、背景がそれより暗くなる方向ではコントラストが上がるだけなので、
どの状況でも読める。基準値は `config/palette.lua` の `reference_background` に置く。

### 後退は明度ではなく彩度で表現する

暗くできない以上、「引っ込んで見える」表現は彩度を落として作る。
マークアップ記号は彩度を元の 30% まで落とした無彩色寄りのグレーとし、明度は基準を満たす位置まで上げる。
本文とのコントラスト比は 1.47 となり、十分に後退して見える。

### 後退した色どうしが同化していないことを確かめる

彩度で後退を表現し、輝度を tier の下限まで持ち上げると、下限に張り付いた色どうしが同化しうる。
コントラスト検査は各色が下限を超えるかしか見ないため、この同化を検出できない。

そこで別の不変条件を置く。palette の中で相異なる hex は、互いに知覚可能な距離を持つこと。
距離は OKLab のユークリッド距離 (色差) で測り、閾値には知覚閾値 (JND) を使う。
意図的に同色のトークンは palette 内で同じリテラルを共有するため、
相異なる hex どうしだけを比べれば、区別すべき色だけが対象になる。

### 目標コントラストは役割ごとに決める

本文の前景色 (Mariana の `#d8dee9`) が基準背景に対して 4.44:1 であり、これが事実上の上限になる。
WCAG AA の 4.5:1 を全色に課すと、ほぼ白かパステルしか使えず見出しの色相差が消える。
そこで役割ごとに tier を定義する。tier と目標値は `config/palette.lua` の `minimum_contrast` に置く。

- text: 読む前景 (見出し、強調、git で色が付くファイル名)
- accent: 補助的に読む前景 (リスト、引用、リンク、コードブロック、行番号)
- symbol: 意図的に後退させる記号 (マークアップ記号、URL、gitignored、dotfile)
- decoration: 装飾 (ツリーのインデント線)

### 色相は保ち、輝度だけ持ち上げる

既存パレットの色相配置はユーザーの美的選択なので尊重する。
OKLCh (知覚均等色空間) で色相を固定したまま輝度を二分探索し、
目標コントラストをちょうど満たす最小の輝度を求める。彩度は sRGB のガマットからはみ出す分だけクリップする。

HSL ではなく OKLCh を使うのは、色相ごとに知覚明度がばらつくのを避けるため。
これにより「H2 の黄色と H3 の紫が同じくらい目立つ」ことを数値で担保できる。

## 個別の設計判断

### インラインコードは accent tier に置く

赤 (色相 25 度付近) は明るくすると sRGB のガマットが急激に痩せる。
text tier まで持ち上げると彩度が 0.145 から 0.066 まで落ち、淡いピンクになって「コード = 赤」の意味が失われる。
インラインコードは本文中の短い span で周囲との対比が効くため、accent tier に置いて赤みを残す。

### 見出し 6 の色相を振り直す

H1 (ティール) と H6 (シアン) の色相差が 22 度しかなかった。元のパレットから受け継いだ問題である。
H6 を青側へ振り、6 色すべての隣接色相差を 30 度以上にする。

なお色相だけに情報を載せない原則から、見出しは bold も併用する。

### `@markup.link` を新たに定義する

実測すると、このキャプチャが捕捉するのは URL ではなくリンクの記号 (`[` `]` `(` `)` `!`) だった。
現在は色が未指定で本文色のまま前に出ている。他の記号と同じ MUTED に揃える。

`@markup.link` は他の markup 言語とも共有する汎用キャプチャなので、
既存の `@punctuation.special` などと同じく `@markup.link.markdown_inline` へスコープする。

副作用として、参照リンク `[label][ref]` の `[ref]` は `@markup.link.label` の後に `@markup.link` が
評価されて後勝ちするため、`[ref]` 全体が MUTED になる。実測で確認済みの既知の挙動として受容する。

### ファイラの薄灰は分けて扱う

`NeoTreeDimText` (薄字、`(empty folder)`) と `NeoTreeExpander` (展開マーカー) は状態を伝えるので
symbol tier まで上げる。一方 `NeoTreeIndentMarker` は装飾なので link を外し、decoration tier に留める。
そうしないとツリーの縦線が目立ちすぎる。

### 不透明な面は自前の背景に対して評価する

ステータスラインのように自前の背景色を持つ面は、透過したターミナル背景の上には無い。
Ghostty の `background-opacity` はウィンドウ背景にだけ掛かり、明示的な背景色を持つセルは不透明に描かれる
(`background-opacity-cells` が既定の `false` のとき。Ghostty 1.3.1 のドキュメントで確認)。

したがってこれらの色は `reference_background` ではなく自前の `bg` に対して評価する。
不透明である以上、前景色に WCAG AA の 4.5:1 を課せる。tier とは別の軸として `surfaces` に分けて持つ。

この分離には副次的な効果がある。`background-opacity-cells` を後から有効にすると前提が崩れるが、
前提を palette に書いておけば、そのときテストが赤くなって気づける。

## アーキテクチャ

色を単一の真実にし、写像を役割ごとに分ける。

| ファイル | 責務 |
| --- | --- |
| `home/.config/nvim/lua/config/palette.lua` | 基準背景、tier ごとの目標コントラスト、色トークン、不透明な面。副作用なし |
| `home/.config/nvim/lua/config/markdown.lua` | treesitter キャプチャからハイライトへの写像 |
| `home/.config/nvim/lua/config/neotree.lua` | NeoTree グループからハイライトへの写像 |
| `home/.config/nvim/lua/config/appearance.lua` | base (Normal / LineNr) と `apply()` のオーケストレーション |
| `home/.config/nvim/lua/plugins/lualine.lua` | プラグイン設定。色は `palette.surfaces` を参照する |

`plugins/neo-tree.lua` はプラグイン設定のまま残し、配色だけを `config/neotree.lua` に分ける。
`plugins/lualine.lua` はハイライトグループではなくプラグイン自身の opts に色を渡すため、
写像ファイルには分けず、palette を参照するだけに留める。

同じ色が複数のトークンに現れる場合 (打ち消し線、未チェック、gitignored はすべて同色) は、
palette 内のローカル変数に束ねてトークンへ割り当てる。トークン名は色ではなく役割で付ける。

`colors` は `{ hex, tier }` の対で持ち、写像側が使う平坦な `hex` テーブルをそこから導出する。
tier を色と同じ場所に持つことで、色を足して tier を書き忘れる事故を構造的に防ぐ。

## 検証戦略

### コントラストをテストで固定する

`scripts/tests/nvim-markdown-probe.lua` が `config.palette` を require し、
全トークンの実際のコントラスト比を計算する。tier の目標を満たさないトークンを列挙して出力し、
bats 側でそれが空であることを検査する。

これにより、後から手で色を変えても検査が守る。目標値も基準背景も palette.lua にしか書かないため、
散文と定義の二重管理による drift が起きない。

### 区別可能性をテストで固定する

同じ probe が相異なる hex の総当たりで OKLab の色差を計算し、JND を下回る組を列挙する。
bats 側でそれが空であることを検査する。

色差の計算は白と黒でちょうど 1.0、同色どうしで 0.0 になる。この 2 点を較正として先に固定する。
負のケースには、実際に同化していた組 (dotfile と旧 RECEDED、色差 0.0049) を sentinel として使う。

### 不透明な面のコントラストをテストで固定する

`surfaces` の各面について、前景色を自前の背景色に対して評価し、
`surface_minimum_contrast` を下回る面を列挙して bats 側で空であることを検査する。

あわせて `plugins/lualine.lua` が palette と同じ値を使っていることを検査する (drift ガード)。
palette を参照していること自体は ast-grep のルールが構文レベルで保証するため、
probe 側は値の一致だけを見る。この 2 つで、値の drift とハードコードの再発の両方を塞ぐ。

負のケース (目標未達の色を注入すると赤になること) を変異注入で確認する。

### ハードコードを ast-grep で禁止する

写像ファイルに hex リテラルを書くことを構文レベルで禁止する。
`kind: string` に regex 制約を掛けるため、根拠として実測値をコメントに書くことは妨げない。
grep では実現できない区別である。

適用範囲は nvim の Lua 全体とし、`palette.lua` だけを除外する。

### neo-tree のグループ名を検証する

neo-tree のハイライトグループ名は treesitter のキャプチャではないため、
タイポしても Neovim は黙って無視する。
neo-tree のソースが見つかる環境では、定義したグループ名がすべて実在することを検査する。
CI にはプラグインを入れないので、その場合は skip する。

### 実環境で目視する

配色は最終的に人間の目でしか判定できない。実際の nvim で README.md と neo-tree を開き、
背後に明るいウィンドウと暗いウィンドウを置いた両方の状態で確認する。

## スコープ外

以下は今回変更しない。

- `@markup.heading.2` と `@markup.strong` は既に目標を満たしているため据え置く
- `@markup.italic` は fg を持たず italic だけで表現する意図的な設計なので変更しない
- `NeoTreeFadeText1` / `NeoTreeFadeText2` はフィルタ表示でのみ使われるため据え置く
- neo-tree の `Directory` / `GitModified` / `GitAdded` / `GitDeleted` は基準を満たすため触らない
- `NeoTreeIgnored` と `NeoTreeWindowsHidden` は `NeoTreeDotfile` への link なので修正を自動で継承する

## 別タスクとして切り出すもの

`~/.config/ghostty/config` は dotfiles の symlink 管理下にない。
つまり Mariana テーマと `background-opacity = 0.8` という前提は他のマシンで再現されない。
palette.lua にはこの前提をコメントで明記するが、Ghostty 設定を dotfiles 管理下へ入れるかどうかは
別タスクとする。
